# AI Agent Game Lab Project Plan

## 1. 프로젝트 방향

초등학생이 자연어로 AI 에이전트에게 요청하여 미니게임의 최초 설정을 함께 만들고, 이후 설정값을 수정하며, 실행 결과를 확인해 프로그램 구조를 이해하도록 돕는 웹앱을 만든다.

핵심 원칙은 다음과 같다.

- AI는 게임 코드를 직접 수정하지 않는다.
- AI는 학생의 자연어 요청을 허용된 settings 변경값으로만 변환한다.
- 실제 게임은 미리 구현된 템플릿과 검증된 설정값을 기반으로 작동한다.
- 게임의 핵심 규칙은 고정하고, 캐릭터/배경/장애물/아이템 같은 표현 요소와 일부 난이도 값만 바꾼다.
- 학생은 선택지뿐 아니라 자유 입력으로도 원하는 표현을 말할 수 있다.
- AI는 학생의 자유 입력을 제한된 visual attributes로 안전하게 매핑한다.
- 학생은 AI 제안을 바로 적용하지 않고, 바뀔 값을 확인한 뒤 적용한다.
- 수업 목표는 게임 제작 자체보다 변수, 조건, 반복, 상태 변화, 테스트와 수정 과정을 경험하는 것이다.

프로젝트의 경험 목표는 다음과 같다.

```text
AI와 함께 게임 만들기 대화
→ 최초 변수와 표현 설정
→ 설정 제출
→ 게임 실행
→ AI에게 수정 요청
→ 변경 전후 확인
→ 다시 실험
```

## 2. 1차 MVP 범위

처음 버전은 DB, 로그인, 실제 AI API 없이 로컬용으로 만든다.

1차 MVP에 포함할 기능:

- Next.js + TypeScript + Tailwind 기반 웹앱
- `/create` 게임 만들기 대화 화면
- `/play` VS Code 느낌의 4분할 작업 화면
- 쿠키런 스타일 러닝 게임 1종
- 게임 만들기 페이즈
- AI 질문에 답하면서 최초 settings 구성
- settings 객체 기반 게임 변경
- 오른쪽 변수/설정 확인창
- mock AI 채팅
- AI 변경 제안 미리보기
- 적용하기, 되돌리기, 기본값 초기화
- 요청 예시 버튼
- 해본 결과 작성
- 변경 기록 localStorage 저장

1차 MVP에서 제외할 기능:

- Supabase DB
- 학생/교사 로그인
- 실제 OpenAI 또는 Gemini API 연결
- 슈팅 게임
- CSV 다운로드
- 교사용 통계 대시보드
- 외부 이미지 업로드
- AI 코드 생성
- 규칙 자체를 바꾸는 자유 게임 제작

## 3. 화면 구조

기본 화면은 두 페이지로 나눈다.

```text
/create
AI와 대화하면서 최초 게임 설정 만들기

→ 설정 제출 / 게임 만들기

/play
완성된 게임을 실행하고 AI로 수정/실험하기
```

### 3.1 `/create`: 게임 만들기 대화 화면

처음 접속하면 완성된 게임이나 4분할 작업실을 바로 보여주지 않는다. 학생이 최초 구상에 집중할 수 있도록 AI 질문과 답변 중심의 단일 화면으로 시작한다.

```text
┌──────────────────────────────────────────────┐
│ AI 게임 만들기                                │
│ 진행: 캐릭터 → 배경 → 장애물 → 아이템 → 난이도 │
├──────────────────────────────────────────────┤
│ AI: 캐릭터는 어떤 모습이면 좋을까요?          │
│                                              │
│ [고양이] [로봇] [토끼] [공룡]                 │
│ 직접 쓰기: [ 소                          ]   │
│                                              │
│ 지금까지 정한 것                             │
│ 캐릭터: 달리는 소                            │
│ 배경: 아직 안 정함                           │
│ 장애물: 아직 안 정함                         │
│ 아이템: 아직 안 정함                         │
│ 난이도: 아직 안 정함                         │
│                                              │
│ [다음]                                       │
└──────────────────────────────────────────────┘
```

AI 질문 예시:

```text
캐릭터는 어떤 모습이면 좋을까요?

[고양이] [로봇] [토끼] [공룡]
직접 쓰기: [소]
```

모든 질문에 답하면 최종 확인 화면을 보여준다.

```text
게임 설정이 완성됐어요.

캐릭터: 달리는 소
배경: 화산
장애물: 바나나 껍질
아이템: 별
난이도: 쉬움

[게임 만들기]
```

학생이 `게임 만들기`를 누르면 최초 settings를 저장하고 `/play`로 이동한다.

### 3.2 `/play`: 게임 실험 작업실

최초 설정이 완성된 뒤에만 4분할 작업실 화면으로 진입한다. `/play`는 게임 실행, 변수 확인, AI 수정 요청, 해본 결과 기록에 집중한다.

```text
┌──────────────────────────────────────────────┐
│ 상단 바: 프로젝트명 / 게임 선택 / 실행 / 저장 │
├──────────────────────────┬───────────────────┤
│                          │ 변수/설정 확인창   │
│ 실제 게임 화면            │ 변경 전후 비교     │
│ 전체 화면의 약 55~60%     │ 학습 개념 표시     │
│                          ├───────────────────┤
│                          │ AI 채팅창          │
│                          │ 요청/응답/적용     │
├──────────────────────────┴───────────────────┤
│ 해본 결과 / 변경 기록                         │
└──────────────────────────────────────────────┘
```

권장 비율:

- 왼쪽 게임 화면: 전체 중앙 영역의 55~60%
- 오른쪽 패널: 전체 중앙 영역의 40~45%
- 오른쪽 위: 변수/설정 확인창 55%
- 오른쪽 아래: AI 채팅창 45%
- 하단: 해본 결과와 변경 기록

## 4. 주요 UI 영역

### 4.1 게임 화면

- 왼쪽 큰 영역에 실제 러닝 게임을 표시한다.
- Canvas 또는 Phaser 기반으로 구현한다.
- HUD에는 점수, 체력, 남은 시간, 현재 테마를 표시한다.
- 실행, 일시정지, 다시 시작 버튼을 제공한다.
- `/create`가 완료되기 전에는 이 화면을 보여주지 않는다.

### 4.2 변수/설정 확인창

`/play` 오른쪽 위에 현재 설정값과 AI가 제안한 변경값을 보여준다.

`/create`에서는 별도의 "지금까지 정한 것" 요약 영역에 최초 settings가 채워지는 과정을 보여준다. `/create`의 요약 영역은 학생용 이름 중심으로 단순하게 보여주고, `/play`의 변수/설정 확인창은 내부 필드와 학습 개념까지 함께 보여준다.

개발자 용어는 내부에서만 사용하고, 화면에는 쉬운 용어를 쓴다.

| 내부 필드 | 학생용 표시 | 학습 개념 |
|---|---|---|
| player.name | 캐릭터 이름 | 변수 |
| player.archetype | 캐릭터 종류 | 변수 |
| player.color | 캐릭터 색 | 변수 |
| player.accessory | 캐릭터 장식 | 변수 |
| player.speed | 캐릭터 속도 | 변수 |
| player.hp | 체력 | 상태 |
| player.jumpPower | 점프 힘 | 변수 |
| background.theme | 배경 분위기 | 변수 |
| background.decoration | 배경 장식 | 변수 |
| obstacle.name | 장애물 이름 | 변수 |
| obstacle.shape | 장애물 모양 | 반복 |
| obstacle.speed | 장애물 속도 | 반복/난이도 |
| obstacle.spawnRate | 장애물 나오는 간격 | 반복 |
| item.name | 아이템 이름 | 변수 |
| item.shape | 아이템 모양 | 반복 |
| item.spawnRate | 아이템 나오는 간격 | 반복 |
| game.duration | 게임 시간 | 조건 |
| game.goalScore | 목표 점수 | 조건 |

변경 제안은 아직 적용되지 않은 상태로 표시한다. 예:

```text
배경: 숲 -> 우주
장애물 나오는 간격: 1200 -> 1800
```

### 4.3 AI 채팅창

`/play` 오른쪽 아래에 배치한다.

`/create`의 AI 대화는 별도 페이지에서 처리한다. `/play`의 AI 채팅창은 생성된 게임을 수정하고 실험하는 용도로만 사용한다.

`/create` 학생 흐름:

1. AI가 캐릭터, 배경, 장애물, 아이템, 난이도를 차례로 질문한다.
2. 학생은 예시 버튼을 누르거나 직접 답한다.
3. AI는 답변을 안전한 settings 값으로 변환한다.
4. "지금까지 정한 것" 요약에 최초 설정이 채워진다.
5. 모든 질문이 끝나면 최종 설정 확인 화면을 보여준다.
6. 학생이 게임 만들기 버튼을 누르면 `/play`로 이동한다.

`/play` 학생 흐름:

1. 자연어로 요청한다.
2. AI가 변경 제안을 만든다.
3. 학생이 변수/설정 확인창에서 바뀔 값을 본다.
4. 적용 버튼을 누른다.
5. 게임을 실행해 확인한다.

초기 요청 예시 버튼:

- 배경을 우주로 바꿔줘
- 캐릭터를 빠르게 해줘
- 장애물이 천천히 나오게 해줘
- 점프를 더 높게 해줘
- 게임 시간을 60초로 해줘

만들기 질문 예시:

- 캐릭터는 어떤 모습이면 좋을까요?
- 배경은 어떤 곳이면 좋을까요?
- 장애물은 무엇이 나오면 좋을까요?
- 아이템은 무엇을 모으면 좋을까요?
- 게임은 쉽게, 보통, 어렵게 중 어떤 느낌이면 좋을까요?

### 4.4 해본 결과

"테스트 로그"라는 이름은 사용하지 않는다. 학생에게는 "해본 결과" 또는 "실험 기록"으로 표시한다.

기록 항목:

- 내가 AI에게 부탁한 것
- 바뀐 것
- 게임을 해보니 어땠나요?
- 다음에 바꾸고 싶은 것

이 데이터는 나중에 연구 자료와 교사용 화면에서 활용할 수 있다.

## 5. 게임 디자인 방향

미니게임은 조잡한 도형만으로 만들지 않고, 오픈 에셋 기반의 정돈된 아케이드 스타일을 목표로 한다.

권장 조합:

- 게임 엔진: Phaser
- 에셋 스타일: Kenney 스타일
- 웹앱 UI: Next.js + Tailwind

디자인 원칙:

- 귀엽지만 너무 유아적이지 않은 수업용 아케이드 스타일
- 설정 변화가 눈에 잘 보여야 한다.
- 캐릭터, 배경, 장애물, 아이템은 자유 입력을 받을 수 있지만 내부에서는 허용된 속성 조합으로 표현한다.
- AI가 이미지를 생성하지 않고, 미리 준비된 에셋 또는 Canvas 조합형 렌더러 중 하나를 선택한다.
- 배경 테마는 색감과 장식으로 명확히 구분한다.
- 학생이 말한 표현이 이름, 색, 장식, 분위기에 반영되어 "내가 AI를 부려서 바꿨다"는 느낌이 나야 한다.

초기 테마 후보:

- 우주
- 숲
- 사막
- 바다
- 학교

초기 요소 후보:

- 캐릭터: 고양이, 로봇, 토끼
- 장애물: 선인장, 돌, 구름
- 아이템: 별, 코인, 하트

자유 입력 처리 예시:

| 학생 입력 | 내부 처리 예시 |
|---|---|
| 소 | animal + brown + horns + tail |
| 상어 | animal + blue + fin |
| 피카츄 | animal + yellow + lightning |
| 용 | fantasy + green + wings + tail |
| 자동차 | vehicle + red + wheels |
| 햄버거 | food + round + yellow/brown |
| 귀신 | monster + white + floating |

상용 캐릭터 이름은 직접 재현하지 않고 일반화한다.

예:

```text
학생 입력: 피카츄
AI 처리: 노란 번개 캐릭터
```

## 6. 게임 구현 방향

1차 게임은 쿠키런 스타일 러닝 게임으로 만든다.

고정할 기본 규칙:

- 캐릭터는 화면 왼쪽에 고정된다.
- 배경과 바닥, 장애물이 왼쪽으로 이동한다.
- 학생은 점프 또는 슬라이드로 장애물을 피한다.
- 아이템을 먹으면 점수가 오른다.
- 장애물에 부딪히면 체력이 줄어든다.
- 제한 시간 또는 목표 점수 도달 시 게임이 끝난다.

사용자가 바꿀 수 있는 범위:

- 캐릭터 모습
- 캐릭터 이름
- 캐릭터 색
- 캐릭터 장식
- 배경 분위기
- 배경 장식
- 장애물 이름과 모양
- 아이템 이름과 모양
- 속도, 점프력, 등장 간격
- 게임 시간
- 목표 점수
- 화면 분위기

규칙을 크게 벗어나는 요청은 직접 반영하지 않고 가능한 설정 변경으로 안내한다.

예:

```text
학생: 총을 쏘게 해줘.
AI: 이 러닝 게임에서는 점프와 슬라이드로 피하는 규칙은 그대로 둘게요.
    대신 장애물을 더 느리게 나오게 하거나 캐릭터를 더 빠르게 만들 수 있어요.
```

settings와 연결할 값:

```ts
player.name
player.archetype
player.color
player.accessory
player.speed
player.hp
player.jumpPower
background.theme
background.mood
background.decoration
obstacle.name
obstacle.type
obstacle.shape
obstacle.speed
obstacle.spawnRate
item.name
item.type
item.shape
item.spawnRate
game.duration
game.goalScore
```

## 7. AI 동작 방식

AI는 두 단계로 작동한다.

### 7.1 게임 만들기 에이전트

게임 만들기 모드에서는 AI가 학생에게 질문하면서 최초 settings를 구성한다.

질문 순서:

1. 캐릭터
2. 배경
3. 장애물
4. 아이템
5. 난이도

각 질문은 예시 선택지와 자유 입력을 함께 제공한다.

```text
캐릭터는 어떤 모습이면 좋을까요?

[고양이] [로봇] [토끼] [공룡]
직접 쓰기: [          ]
```

학생의 자유 입력은 제한된 속성으로 매핑한다.

```ts
type PlayerArchetype =
  | "animal"
  | "robot"
  | "vehicle"
  | "monster"
  | "fantasy"
  | "food"
  | "custom";

type PlayerAccessory =
  | "none"
  | "ears"
  | "horns"
  | "tail"
  | "wings"
  | "helmet"
  | "lightning"
  | "hat"
  | "fin";
```

만들기 응답 예시:

```json
{
  "field": "player",
  "changes": {
    "player": {
      "name": "달리는 소",
      "archetype": "animal",
      "color": "brown",
      "accessory": "horns"
    }
  },
  "studentExplanation": "갈색 몸통과 뿔이 있는 달리는 소로 만들게요."
}
```

### 7.2 게임 수정 에이전트

게임 실험 모드에서는 AI가 전체 settings를 반환하지 않고, 변경 patch만 반환한다.

권장 응답 형식:

```json
{
  "changes": {
    "background": "space",
    "player": {
      "name": "번개 고양이",
      "color": "yellow"
    }
  },
  "studentExplanation": "배경을 우주로 바꾸고 캐릭터 이름과 색을 바꾸었어요."
}
```

서버 또는 로컬 검증 함수가 기존 settings에 changes를 병합한다.

중요 규칙:

- 허용되지 않은 필드는 제거한다.
- 허용되지 않은 문자열은 기본값으로 대체한다.
- 수치가 범위를 벗어나면 허용 범위 안으로 조정한다.
- 학생 요청과 무관한 값을 임의로 바꾸지 않는다.
- AI 제안은 바로 적용하지 않고 미리보기 상태로 둔다.
- 게임 규칙을 바꾸는 요청은 표현 요소나 난이도 조정으로 대체 안내한다.

1차 MVP에서는 실제 AI API 대신 mock agent를 사용한다.

## 8. 로컬 저장 방식

1차 MVP는 localStorage를 사용한다.

저장할 데이터:

- `/create`에서 만든 최초 settings
- 현재 settings
- 적용 이력
- AI 요청 기록
- 해본 결과 기록

나중에 Supabase를 붙일 때 localStorage 저장 계층을 API 저장 계층으로 교체한다.

## 9. 교사용 화면 구상

교사용 화면은 2차 이후 기능이다.

목적:

- 학생별 진행 상태 확인
- 학생의 AI 요청 기록 확인
- 설정값 변경 이력 확인
- 해본 결과와 성찰 확인
- 최종 산출물 확인
- 연구용 데이터 다운로드

초기 로컬 버전에서는 같은 브라우저의 localStorage 데이터를 보여주는 간단한 `/teacher` 페이지로 시작할 수 있다.

DB 연결 후에는 여러 학생의 데이터를 모아서 보는 구조로 확장한다.

교사용 화면 예시:

```text
학생 목록
→ 학생 선택
→ 프로젝트 종류
→ 최종 설정
→ AI 요청 기록
→ 변경된 변수
→ 해본 결과
→ 최종 게임 미리보기
```

## 10. 개발 단계

### 1단계: 프로젝트 기반

- Next.js + TypeScript + Tailwind 프로젝트 생성
- 기본 레이아웃과 디자인 토큰 구성
- `/create` 대화형 생성 화면 구현
- `/play` 화면 4분할 UI 구현

### 2단계: settings 구조

- 게임 settings 타입 정의
- 기본 settings 작성
- 규칙 설정과 표현 설정 분리
- 자유 입력을 visual attributes로 매핑하는 함수 작성
- patch 병합 함수 작성
- 검증 함수 작성

### 3단계: 게임 만들기 페이즈

- AI 질문 흐름 구현
- 예시 선택지 + 자유 입력 UI 구현
- 학생 답변을 최초 settings로 반영
- `/create` 요약 영역에 생성 중 settings 표시
- 질문 완료 후 최종 설정 확인 화면 구현
- 제출 시 localStorage에 최초 settings 저장
- 제출 후 `/play`로 이동

### 4단계: 러닝 게임 MVP

- Phaser 또는 Canvas 기반 러닝 게임 구현
- `/create`에서 생성된 settings로 게임 시작
- settings 값으로 캐릭터, 배경, 장애물, 아이템, 난이도 반영
- 조합형 캐릭터 렌더러 구현
- 점수, 체력, 시간 HUD 구현

### 5단계: mock AI 수정

- 자연어 요청을 changes patch로 바꾸는 mock 함수 작성
- AI 채팅 UI 연결
- 변경 제안 미리보기 구현
- 적용하기, 되돌리기, 기본값 초기화 구현

### 6단계: 기록 기능

- 해본 결과 작성 UI 구현
- 변경 이력 표시
- localStorage 저장

### 7단계: 확장

- 실제 AI API 연결
- 슈팅 게임 추가
- Supabase 저장
- 교사용 화면
- CSV 또는 JSON 내보내기

## 12. 1차 성공 기준

1차 MVP가 성공했다고 볼 기준:

- 처음에는 `/create` 대화 화면으로 시작한다.
- `/create`에서는 게임 작업실을 보여주지 않고 AI 질문과 답변에 집중할 수 있다.
- 학생이 AI 질문에 답하면서 최초 게임 설정을 만들 수 있다.
- 선택지와 자유 입력 모두 최초 settings에 반영된다.
- 자유 입력은 허용된 캐릭터/배경/장애물/아이템 속성으로 안전하게 매핑된다.
- 질문 완료 후 최종 설정 확인 화면에서 게임 만들기를 누르면 `/play`로 이동한다.
- `/play` 화면이 작업실 구조로 안정적으로 보인다.
- 러닝 게임이 정상 실행된다.
- settings 값을 바꾸면 게임에 반영된다.
- mock AI 요청으로 변경 제안이 만들어진다.
- 학생이 적용 전 바뀔 값을 확인할 수 있다.
- 적용 후 게임이 바뀐다.
- 되돌리기와 기본값 초기화가 가능하다.
- 학생이 해본 결과를 작성할 수 있다.
- 기록이 localStorage에 남는다.

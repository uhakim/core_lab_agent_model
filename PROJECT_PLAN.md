# AI Agent Game Lab Project Plan

## 1. 프로젝트 방향

초등학생이 자연어로 AI 에이전트에게 요청하여 미니게임의 설정값을 바꾸고, 실행 결과를 확인하며 프로그램 구조를 이해하도록 돕는 웹앱을 만든다.

핵심 원칙은 다음과 같다.

- AI는 게임 코드를 직접 수정하지 않는다.
- AI는 학생의 자연어 요청을 허용된 settings 변경값으로만 변환한다.
- 실제 게임은 미리 구현된 템플릿과 검증된 설정값을 기반으로 작동한다.
- 학생은 AI 제안을 바로 적용하지 않고, 바뀔 값을 확인한 뒤 적용한다.
- 수업 목표는 게임 제작 자체보다 변수, 조건, 반복, 상태 변화, 테스트와 수정 과정을 경험하는 것이다.

## 2. 1차 MVP 범위

처음 버전은 DB, 로그인, 실제 AI API 없이 로컬용으로 만든다.

1차 MVP에 포함할 기능:

- Next.js + TypeScript + Tailwind 기반 웹앱
- VS Code 느낌의 4분할 작업 화면
- 쿠키런 스타일 러닝 게임 1종
- settings 객체 기반 게임 변경
- 오른쪽 변수/설정 확인창
- mock AI 채팅
- AI 변경 제안 미리보기
- 적용하기, 되돌리기, 기본값 초기화
- 요청 예시 버튼
- 해본 결과 작성
- 변경 기록 localStorage 저장

1차 MVP에서 제외할 기능:

- Supabase DB
- 학생/교사 로그인
- 실제 OpenAI 또는 Gemini API 연결
- 슈팅 게임
- CSV 다운로드
- 교사용 통계 대시보드
- 외부 이미지 업로드
- AI 코드 생성

## 3. 화면 구조

기본 화면은 작업실 형태로 구성한다.

```text
┌──────────────────────────────────────────────┐
│ 상단 바: 프로젝트명 / 게임 선택 / 실행 / 저장 │
├──────────────────────────┬───────────────────┤
│                          │ 변수/설정 확인창   │
│ 실제 게임 화면            │ 변경 전후 비교     │
│ 전체 화면의 약 55~60%     │ 학습 개념 표시     │
│                          ├───────────────────┤
│                          │ AI 채팅창          │
│                          │ 요청/응답/적용     │
├──────────────────────────┴───────────────────┤
│ 해본 결과 / 변경 기록                         │
└──────────────────────────────────────────────┘
```

권장 비율:

- 왼쪽 게임 화면: 전체 중앙 영역의 55~60%
- 오른쪽 패널: 전체 중앙 영역의 40~45%
- 오른쪽 위: 변수/설정 확인창 55%
- 오른쪽 아래: AI 채팅창 45%
- 하단: 해본 결과와 변경 기록

## 4. 주요 UI 영역

### 4.1 게임 화면

- 왼쪽 큰 영역에 실제 러닝 게임을 표시한다.
- Canvas 또는 Phaser 기반으로 구현한다.
- HUD에는 점수, 체력, 남은 시간, 현재 테마를 표시한다.
- 실행, 일시정지, 다시 시작 버튼을 제공한다.

### 4.2 변수/설정 확인창

오른쪽 위에 현재 설정값과 AI가 제안한 변경값을 보여준다.

개발자 용어는 내부에서만 사용하고, 화면에는 쉬운 용어를 쓴다.

| 내부 필드 | 학생용 표시 | 학습 개념 |
|---|---|---|
| player.name | 캐릭터 이름 | 변수 |
| player.color | 캐릭터 색 | 변수 |
| player.speed | 캐릭터 속도 | 변수 |
| player.hp | 체력 | 상태 |
| player.jumpPower | 점프 힘 | 변수 |
| background | 배경 | 변수 |
| obstacle.speed | 장애물 속도 | 반복/난이도 |
| obstacle.spawnRate | 장애물 나오는 간격 | 반복 |
| item.spawnRate | 아이템 나오는 간격 | 반복 |
| game.duration | 게임 시간 | 조건 |
| game.goalScore | 목표 점수 | 조건 |

변경 제안은 아직 적용되지 않은 상태로 표시한다. 예:

```text
배경: 숲 -> 우주
장애물 나오는 간격: 1200 -> 1800
```

### 4.3 AI 채팅창

오른쪽 아래에 배치한다.

학생 흐름:

1. 자연어로 요청한다.
2. AI가 변경 제안을 만든다.
3. 학생이 변수/설정 확인창에서 바뀔 값을 본다.
4. 적용 버튼을 누른다.
5. 게임을 실행해 확인한다.

초기 요청 예시 버튼:

- 배경을 우주로 바꿔줘
- 캐릭터를 빠르게 해줘
- 장애물이 천천히 나오게 해줘
- 점프를 더 높게 해줘
- 게임 시간을 60초로 해줘

### 4.4 해본 결과

"테스트 로그"라는 이름은 사용하지 않는다. 학생에게는 "해본 결과" 또는 "실험 기록"으로 표시한다.

기록 항목:

- 내가 AI에게 부탁한 것
- 바뀐 것
- 게임을 해보니 어땠나요?
- 다음에 바꾸고 싶은 것

이 데이터는 나중에 연구 자료와 교사용 화면에서 활용할 수 있다.

## 5. 게임 디자인 방향

미니게임은 조잡한 도형만으로 만들지 않고, 오픈 에셋 기반의 정돈된 아케이드 스타일을 목표로 한다.

권장 조합:

- 게임 엔진: Phaser
- 에셋 스타일: Kenney 스타일
- 웹앱 UI: Next.js + Tailwind

디자인 원칙:

- 귀엽지만 너무 유아적이지 않은 수업용 아케이드 스타일
- 설정 변화가 눈에 잘 보여야 한다.
- 캐릭터, 배경, 장애물, 아이템은 허용된 목록 안에서만 바뀐다.
- AI가 이미지를 생성하지 않고, 미리 준비된 에셋 중 하나를 선택한다.
- 배경 테마는 색감과 장식으로 명확히 구분한다.

초기 테마 후보:

- 우주
- 숲
- 사막
- 바다
- 학교

초기 요소 후보:

- 캐릭터: 고양이, 로봇, 토끼
- 장애물: 선인장, 돌, 구름
- 아이템: 별, 코인, 하트

## 6. 게임 구현 방향

1차 게임은 쿠키런 스타일 러닝 게임으로 만든다.

기본 규칙:

- 캐릭터는 화면 왼쪽에 고정된다.
- 배경과 바닥, 장애물이 왼쪽으로 이동한다.
- 학생은 점프 또는 슬라이드로 장애물을 피한다.
- 아이템을 먹으면 점수가 오른다.
- 장애물에 부딪히면 체력이 줄어든다.
- 제한 시간 또는 목표 점수 도달 시 게임이 끝난다.

settings와 연결할 값:

```ts
player.name
player.color
player.speed
player.hp
player.jumpPower
background
obstacle.type
obstacle.speed
obstacle.spawnRate
item.type
item.spawnRate
game.duration
game.goalScore
```

## 7. AI 동작 방식

AI는 전체 settings를 반환하지 않고, 변경 patch만 반환한다.

권장 응답 형식:

```json
{
  "changes": {
    "background": "space",
    "player": {
      "name": "번개 고양이",
      "color": "yellow"
    }
  },
  "studentExplanation": "배경을 우주로 바꾸고 캐릭터 이름과 색을 바꾸었어요."
}
```

서버 또는 로컬 검증 함수가 기존 settings에 changes를 병합한다.

중요 규칙:

- 허용되지 않은 필드는 제거한다.
- 허용되지 않은 문자열은 기본값으로 대체한다.
- 수치가 범위를 벗어나면 허용 범위 안으로 조정한다.
- 학생 요청과 무관한 값을 임의로 바꾸지 않는다.
- AI 제안은 바로 적용하지 않고 미리보기 상태로 둔다.

1차 MVP에서는 실제 AI API 대신 mock agent를 사용한다.

## 8. 로컬 저장 방식

1차 MVP는 localStorage를 사용한다.

저장할 데이터:

- 현재 settings
- 적용 이력
- AI 요청 기록
- 해본 결과 기록

나중에 Supabase를 붙일 때 localStorage 저장 계층을 API 저장 계층으로 교체한다.

## 9. 교사용 화면 구상

교사용 화면은 2차 이후 기능이다.

목적:

- 학생별 진행 상태 확인
- 학생의 AI 요청 기록 확인
- 설정값 변경 이력 확인
- 해본 결과와 성찰 확인
- 최종 산출물 확인
- 연구용 데이터 다운로드

초기 로컬 버전에서는 같은 브라우저의 localStorage 데이터를 보여주는 간단한 `/teacher` 페이지로 시작할 수 있다.

DB 연결 후에는 여러 학생의 데이터를 모아서 보는 구조로 확장한다.

교사용 화면 예시:

```text
학생 목록
→ 학생 선택
→ 프로젝트 종류
→ 최종 설정
→ AI 요청 기록
→ 변경된 변수
→ 해본 결과
→ 최종 게임 미리보기
```

## 10. 개발 단계

### 1단계: 프로젝트 기반

- Next.js + TypeScript + Tailwind 프로젝트 생성
- 기본 레이아웃과 디자인 토큰 구성
- `/play` 화면 4분할 UI 구현

### 2단계: settings 구조

- 게임 settings 타입 정의
- 기본 settings 작성
- patch 병합 함수 작성
- 검증 함수 작성

### 3단계: 러닝 게임 MVP

- Phaser 또는 Canvas 기반 러닝 게임 구현
- settings 값으로 캐릭터, 배경, 장애물, 아이템, 난이도 반영
- 점수, 체력, 시간 HUD 구현

### 4단계: mock AI

- 자연어 요청을 changes patch로 바꾸는 mock 함수 작성
- AI 채팅 UI 연결
- 변경 제안 미리보기 구현
- 적용하기, 되돌리기, 기본값 초기화 구현

### 5단계: 기록 기능

- 해본 결과 작성 UI 구현
- 변경 이력 표시
- localStorage 저장

### 6단계: 확장

- 실제 AI API 연결
- 슈팅 게임 추가
- Supabase 저장
- 교사용 화면
- CSV 또는 JSON 내보내기

## 12. 1차 성공 기준

1차 MVP가 성공했다고 볼 기준:

- `/play` 화면이 작업실 구조로 안정적으로 보인다.
- 러닝 게임이 정상 실행된다.
- settings 값을 바꾸면 게임에 반영된다.
- mock AI 요청으로 변경 제안이 만들어진다.
- 학생이 적용 전 바뀔 값을 확인할 수 있다.
- 적용 후 게임이 바뀐다.
- 되돌리기와 기본값 초기화가 가능하다.
- 학생이 해본 결과를 작성할 수 있다.
- 기록이 localStorage에 남는다.
